# 外部サービス利用申請 仕様書兼要件書

この資料はPhase1で使用します。まずは仕様からテスト観点を洗い出してください。判断に迷う点は、仕様不備としてファシリテーターへ質問して構いません。

## 1. 業務概要

社内利用者はSalesforce画面から外部サービスの利用申請を登録します。

登録された申請はSalesforceには保存せず、外部APIを経由して申請管理用DBに保存します。利用者は同じ画面で最近登録された申請を確認できます。

## 2. 利用者

- 研修参加者は共有Developer Orgへログインして画面を操作します。
- 同時に複数名が同じ画面を利用する想定です。
- 参加者PCからMuleやMySQLへ直接接続することは想定しません。

## 3. 画面構成

画面は次の領域で構成します。

| 領域 | 概要 |
| --- | --- |
| 説明 | この画面の用途を表示する |
| 最近の申請 | 直近の申請を一覧表示する |
| 新規申請 | 申請内容を入力して登録する |

処理中は画面上にspinnerを表示します。

## 4. 最近の申請一覧

一覧には、確認しやすいよう主要項目を表示します。

| 表示項目 | 補足 |
| --- | --- |
| 申請番号 | クリックすると詳細を表示する |
| 申請者名 | 申請者を識別する名前 |
| 法人名 | 個人申請の場合は空になることがある |
| サービスプラン | 選択されたプラン |
| 処理ステータス | 登録後の状態 |
| 優先フラグ | 優先または通常 |
| 登録日時 | 登録された日時 |

`再取得` ボタンで最新の一覧を取得します。並び順、表示件数、同時更新時の見え方について疑問がある場合は確認してください。

## 5. 申請詳細

一覧の申請番号をクリックすると、詳細モーダルを表示します。

詳細では、一覧に出ていない項目も含めて申請内容を確認できます。

想定される詳細項目:

- 申請番号
- 申請者区分
- 申請者名
- 法人名
- メールアドレス
- 連絡方法
- 電話番号
- サービスプラン
- 追加要望有無
- 追加要望詳細
- 備考
- 処理ステータス
- 優先フラグ
- 登録日時

空値の表示方法は画面仕様として確認対象です。

## 6. 新規申請フォーム

| 項目 | 入力形式 | 必須 | 補足 |
| --- | --- | --- | --- |
| 申請者区分 | 選択 | 必須 | 法人または個人 |
| 申請者名 | テキスト | 必須 | 長すぎる名前は受け付けない |
| 法人名 | テキスト | 条件付き | 法人申請で使用する |
| メールアドレス | メール | 必須 | 通知先として扱う |
| 連絡方法 | 選択 | 必須 | メールまたは電話 |
| 電話番号 | テキスト | 条件付き | 電話連絡で使用する |
| サービスプラン | 選択 | 必須 | STANDARDまたはPREMIUM |
| 追加要望有無 | 選択 | 必須 | ありまたはなし |
| 追加要望詳細 | 複数行テキスト | 条件付き | 追加要望ありの場合に使用する |
| 備考 | 複数行テキスト | 任意 | 補足情報 |

条件付き項目は、不要になったタイミングで値がクリアされる想定です。

最大文字数、空白だけの入力、メールアドレスの許容範囲などは、テスト設計時に確認してください。

## 7. 登録

登録ボタンを押すと、入力内容を外部APIへ送信します。

登録に成功した場合:

- 成功メッセージを表示します。
- 入力フォームを初期状態に戻します。
- 最近の申請一覧を再取得します。
- 登録した申請が一覧または詳細で確認できることを期待します。

登録に失敗した場合:

- 利用者が理解できるエラーメッセージを赤いToastで表示します。
- 必要に応じて入力内容を修正して再実行できるようにします。

二重送信を防ぐため、処理中の操作可否は確認対象です。

## 8. 業務ルール

申請内容により、処理ステータスや優先フラグが決まります。

ただし、どの条件で優先扱いになるかは、この資料だけでは完全には確定しません。テスト観点として、どの組み合わせを確認すべきか、ファシリテーターへ質問してください。

少なくとも次の観点は考慮してください。

- 申請者区分の違い
- サービスプランの違い
- 条件付き必須項目の有無
- 任意項目が空の場合
- 一覧・詳細に表示される値

## 9. エラー

次のような失敗が起こり得ます。

- 必須入力が不足している
- 入力形式が正しくない
- 条件付き必須項目が不足している
- 外部APIへ接続できない
- DB処理に失敗する
- 想定外の応答が返る

どの層で検出すべきか、利用者へどのように伝えるべきか、赤いToast以外にも画面上の補助表示が必要かは、テスト観点として整理してください。

## 10. アーキテクチャ上の制約

- LWCからMuleへ直接通信しません。
- ApexはNamed Credential `MuleWorkshop` を使って外部APIを呼び出します。
- Salesforceには申請データを保存しません。
- 更新・削除機能はありません。
- Quick Tunnel URLはソースコードに書きません。

## 11. Phase1で作成するもの

Phase1では、ソースコードを読まずに次を作成します。

- テスト観点一覧
- 確認質問一覧
- 優先度の高いテストケース案
- 仕様だけでは判断できないリスク

## 12. 質問してよい例

- 一覧は最大何件まで表示する想定ですか。
- 登録日時はどのタイムゾーンですか。
- 電話番号の形式に制約はありますか。
- メールアドレスはどこまで厳密に検証しますか。
- 優先扱いになる条件は何ですか。
- DB障害時、画面にはどの程度詳しいエラーを出しますか。
- 複数人が同時に登録した場合、申請番号は重複しませんか。
- 空欄と未入力と空白だけの入力は同じ扱いですか。

## 13. 注意

網羅率が高いことだけで品質が高いとは限りません。業務ルール、境界値、例外時の振る舞い、運用手順、ログや個人情報の扱いも品質の一部として考えてください。
